# Internationalisation (i18n)

## Scope

- **Supported locales:** `en` (default) and `fr` only. No German or URL-prefix routing in this version.
- **Libraries:** `i18next`, `react-i18next`, `i18next-http-backend` (loads JSON from `public/locales/`).
- **Persistence:** `uiLocale` is stored via `getAppStorage()` (`lab:uiLocale`). On startup, `initI18n()` reads this key; if it is not `en` or `fr`, the app uses **English**.

## Bootstrap

1. `src/main.jsx` awaits `initI18n()` before mounting React (so the first paint has the correct language).
2. `src/app/providers.tsx` wraps the app in `I18nextProvider` and syncs `document.documentElement.lang` plus `uiLocale` storage on `languageChanged`.

## Files

| Path                                 | Role                                                                |
| ------------------------------------ | ------------------------------------------------------------------- |
| `src/i18n/index.ts`                  | `initI18n()`, `i18n` instance, `SUPPORTED_LOCALES`, storage helpers |
| `public/locales/en/translation.json` | English UI strings                                                  |
| `public/locales/fr/translation.json` | French UI strings                                                   |
| `public/locales/{en,fr}/pdf.json`    | Reserved for invoice / QR-bill PDF copy (non-React code paths)      |

## Adding strings

1. Add the key to **both** `en` and `fr` `translation.json` files (keep nesting consistent, e.g. `auth.signIn`).
2. In React: `const { t } = useTranslation();` then `t('auth.signIn')`.
3. Avoid new user-visible literals in JSX without `t()` (enforce gradually with ESLint `i18next/no-literal-string` when stable).

## Case workflow status labels

`src/utils/cases/caseStatus.js` defines `CASE_STATUS_OPTIONS` with **`labelKey`** (e.g. `caseStatus.in_fabrication`), not hardcoded English labels. Use `t(option.labelKey)` in components, or `getCaseStatusLabel(uiId)` where the global `i18n` instance is acceptable (e.g. patient formatting after init).

## PDFs and non-React code

For generators that cannot use hooks, use:

```ts
import { i18n } from "@/i18n";

const lng = i18n.language?.startsWith("fr") ? "fr" : "en";
const tPdf = i18n.getFixedT(lng, "pdf");
const line = tPdf("someKey");
```

Populate `public/locales/*/pdf.json` as you migrate invoice / QR strings. **html2canvas** (or similar) captures must run after any `changeLanguage` that affects the preview DOM.

## Title copy and capitalization

- Put **final** English/French wording and casing in **`translation.json`** (Title case or sentence case per language), not hardcoded in panel config objects.
- Overview panel headings use **`overview.panels.<id>.title`**; nav uses **`nav.*`**.

## Finding remaining literals (audit)

- Grep for `placeholder="`, `aria-label="`, `title="` with ASCII letters in `src/components` and `src/pages`.
- Optionally add **`i18next-parser`** or **`eslint-plugin-i18next`** when the codebase is mostly migrated.

## API errors

v1 keeps backend `error` strings as-is; map known codes to `t('errors.*')` at the client boundary (e.g. where `ApiError` / `getApiUserMessage` surfaces `userMessage`).

## Locale defaults

This CRM defaults to **English** and persists locale in browser storage only until URL routing is added.
